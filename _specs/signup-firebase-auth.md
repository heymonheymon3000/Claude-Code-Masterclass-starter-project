# Spec for Signup Firebase Auth

branch: claude/feature/signup-firebase-auth

## Summary
- Wire the existing signup form (`app/(public)/signup/`) to Firebase Authentication using the Firebase Web SDK.
- On successful account creation, generate a random PascalCase codename from three word sets and set it as the user's Firebase display name.
- Create a document in a Firestore `users` collection storing only the user's `codename` and `id` (no email).

## Functional Requirements
- The signup form submits email and password to Firebase Auth's `createUserWithEmailAndPassword`.
- On success, a codename is generated by randomly selecting one word from each of three distinct word sets and joining them in PascalCase (e.g. `SwiftSilentFox`).
- The generated codename is saved to the Firebase Auth user profile as the display name via `updateProfile`.
- A Firestore document is created at `users/{uid}` containing:
  - `codename` — the generated PascalCase codename
  - `id` — the Firebase Auth user's UID
  - No email or other PII is stored.
- On error (e.g. email already in use, weak password), the form surfaces a human-readable error message to the user.
- Only the Firebase Web SDK is used — no server-side code, no Admin SDK.
- After successful signup, the user is redirected to `/heists`.

## Possible Edge Cases
- Firebase may reject the signup if the email is already registered or the password is too short — the form must handle and display these errors.
- Codename generation must guarantee uniqueness within each word set (no repeated word categories) and produce a readable result.
- Network failures during `updateProfile` or Firestore write after a successful `createUserWithEmailAndPassword` call should not leave the user in a broken state — errors should be caught and surfaced.
- The Firestore write must not include the user's email under any circumstances.

## Acceptance Criteria
- Submitting valid email and password creates a Firebase Auth account.
- The new user's Firebase display name is set to a randomly generated PascalCase codename.
- A document exists at `users/{uid}` in Firestore with `codename` and `id` fields only.
- The `users/{uid}` document does NOT contain an email field.
- Invalid input (duplicate email, weak password) shows an appropriate error message in the form.
- Successful signup redirects the user to `/heists`.
- Only the Firebase Web SDK is used throughout.

## Open Questions
- None.

## Testing Guidelines
Create a test file(s) in the ./tests folder for the new feature, and create meaningful tests for the following cases, without going too heavy:
- The codename generator produces a non-empty PascalCase string.
- The codename generator always picks one word from each of the three word sets.
- Two calls to the generator return different results (probabilistic — run a few times).
- `AuthForm` in signup mode calls Firebase `createUserWithEmailAndPassword` on submit.
- On auth success, `updateProfile` is called with the generated codename.
- On auth success, a Firestore document is written with `codename` and `id`, without email.
- On auth failure, an error message is rendered in the form.
